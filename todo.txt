We got this compiler to pass the test suites as much as possible.
Some of the tests where listed as not passing due to the test assuming
a given sizeof(int) or sizeof(long), or sizeof(long long).  We added
compiler switches to specify each size so any code with assumptions
could compile.  This uncovered various long math bugs.

gcc-c-torture (sample of 500): 418 pass (84%); remaining are GCC-specific
  features the Z80 port can't reasonably implement (alloca, &&label /
  computed goto, __builtin_return_address, nested-function trampolines,
  __complex__, packed attributes) plus a handful of `int32plus`-required
  tests that we run under default --int=16.

Codegen / driver fixes landed during this pass:
  - 64-bit ++/-- for long_long (was only updating low 16 bits)
  - condition-test for long_long expressions (was only ORing low 32 bits)
  - emit EXTRN for implicit-external function symbols (un-stuck many
    torture tests that called abort()/exit() without a prototype)
  - rewrite __builtin_memcpy / memset / abort / trap / unreachable /
    malloc / etc. to their libc equivalents
  - latin-1 fallback for non-UTF-8 source bytes
  - __SIZE_TYPE__ / __PTRDIFF_TYPE__ / __WCHAR_TYPE__ predefined macros
  - struct-array element initialized from a compound literal:
    `struct Wrap arr[] = { (struct Wrap){fn}, fn };` was storing the
    *address* of the materialized literal in the first member instead
    of doing a byte-copy.  Manifested as an infinite loop in
    c-testsuite 00216 under --long=64.
